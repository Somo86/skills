# Spec-Driven Development Workflow Example

This example shows how to properly apply a Spec-Driven Development workflow in the correct order.

The example is based on a Trip Planner app, but the same workflow can be used for many different products and use cases.

This guide is adapted for Codex usage. When the original Spec Kit workflow would normally reference generic slash commands, this example translates them into Codex-friendly usage with installed `$speckit-*` skills and repository-local artifacts.

## Example Product

Trip Planner is an application that helps travelers organize multi-city trips with realistic timing, budget guidance, and offline access on mobile devices.

## Workflow Order

Follow the workflow steps in the exact order defined in this document. Do not skip ahead. Each step creates the context needed for the next one.

Before Step 1, establish the project foundation once by defining the governing principles that will guide all later specifications, plans, tasks, implementation decisions, and reviews.

## Workflow Map

0. **Establish Project Principles**
   Define the governing principles for all later work.

1. **Specify**
   Capture problem and scope.

2. **Plan**
   Select tech and architecture.

3. **Break Into Tasks**
   Generate an ordered, testable backlog.

4. **Implement**
   Code in small, reviewed slices.

5. **Tests**
   Verify every requirement continuously.

6. **Maintain**
   Update the specification and refactor safely.

## Codex Quick Reference

- `$speckit-constitution`: create the project constitution and guiding principles
- `$speckit-specify`: create or refine `spec.md`
- `$speckit-plan`: create or refine `plan.md`
- `$speckit-clarify`: resolve underspecified areas after planning
- `$speckit-tasks`: generate `tasks.md`
- `$speckit-analyze`: check consistency and coverage across artifacts
- `$speckit-implement`: implement approved tasks in small slices

## Jira Convention

This workflow is optimized for Jira-based teams.

Before starting a new specification, make sure the user provides a Jira epic or ticket ID such as `JCOM-123`.

Do not start the `Specify` step without that ID.

Preserve the Jira ID in the branch short name so the resulting Spec Kit branch follows this pattern:

```text
001-jcom-123-short-feature-name
```

The numeric prefix is generated by Spec Kit.

The Jira ID must be preserved in the short-name portion of the branch.

When possible, derive the short feature name from the Jira ticket title by using Jira MCP or Jira CLI.

If Jira data is not available, derive the short feature name from the specification request itself.

## Foundation Step: Establish Project Principles

This is a one-time foundation step.

Before starting the feature workflow, define the project principles that will govern code quality, testing expectations, user experience consistency, performance expectations, and other development standards.

In upstream Spec Kit examples, this is expressed with:

```text
/speckit.constitution
```

In Codex, prefer the installed skill:

```text
$speckit-constitution
```

### Goal

Create the governing principles and development guidelines that will guide all subsequent work.

### How to Implement This Step

Ask Codex to generate the project constitution so it can serve as the baseline for all later phases.

Example Codex prompt:

```text
$speckit-constitution

Create principles focused on code quality, testing standards, user experience consistency, and performance requirements.
```

### Expected Output

At the end of this step, the agent should generate the project constitution artifact, typically under the Spec Kit memory area for the repository.

### Review Guidance

Review the constitution carefully before proceeding.

These principles will influence all later specifications, plans, tasks, and implementation choices, so they should reflect the real standards of the project rather than generic defaults.

## Step 1: Specify

### Goal

Write a crisp product spec that states the problem, scope, users, success criteria, and constraints.

Focus on what and why.

Avoid implementation details here.

### Definition

The `Specify` step is where the product intent becomes explicit. This is the foundation for the rest of the workflow. A good specification should make it clear:

- what problem is being solved
- who the product is for
- which user flows matter most
- which constraints must be respected
- what is intentionally out of scope

At this stage, do not describe technical architecture, APIs, database design, frameworks, or implementation tasks. Those belong in later steps.

### Codex Adaptation

In Spec Kit, this step is usually driven through the specification workflow. In this Codex-adapted version, the user should ask Codex to create or refine the specification artifact for the feature or product idea, using the provided business context and constraints.

Example Codex prompt:

```text
Create the specification for a Trip Planner app. Focus on the problem, users, core flows, constraints, success criteria, and out-of-scope items. Keep it product-focused and avoid implementation details.
```

### How to Implement This Step

At this point, the goal is to capture the specification in a project-local artifact that lives in version control, so both humans and agents can reference the same source of truth.

Before starting this step for a new feature, ask the user for the Jira epic or ticket ID if they have not already provided it.

Do not proceed without that value.

Then try to retrieve the Jira ticket title and use it to form the short feature name.

If that lookup fails, use a concise name derived from the specification request.

In upstream Spec Kit examples, this is often shown as `/specify`.

In Codex, prefer the installed skill:

```text
$speckit-specify
```

For this Codex-based workflow, adapt that into a direct request to Codex that asks it to create the specification artifact in the repository.

Also make sure the Jira ID is preserved in the feature branch short name.

Example Codex prompt:

```text
$speckit-specify JCOM-123 Build a free-to-use Trip Planner platform.

Preserve the Jira ID in the branch short name.
Use the Jira ticket title to derive the short feature name when available.
Use branch short name: jcom-123-trip-planner.

Create a project-local specification and save it in the repository so it can be version-controlled and referenced by both humans and agents.

Build a trip planner that generates day-by-day itineraries for multi-city travel.
Include personas, key flows, success metrics, and constraints related to performance, privacy, and offline usage.

Problem: help travelers plan multi-city trips with realistic timing, budget guidance, and offline availability.
Users: casual travelers, travel bloggers, and small tour operators.
Key flows: create trips, add cities, auto-generate itinerary, adjust by preferences such as pace, interests, and budget, export to mobile.
Non-functional constraints: P95 itinerary generation under 4 seconds for 7-day trips. Store PII securely. Offline read mode on mobile.
Out of scope: airline booking, hotel payments.
```

This prompt keeps the step aligned with Spec Kit intent while making it usable through Codex.

The important point is not the exact phrasing of the prompt, but the sequence:

1. ask for the Jira epic or ticket ID
2. retrieve the Jira title when possible
3. preserve the Jira ID in the branch short name
4. define the product intent
5. capture it in a local specification artifact
6. keep that artifact under version control before moving to the next workflow step

### Expected Output

At the end of this step, the agent will generate an initial `spec.md` file based on your request.

The more context and detail you provide, the stronger that first draft will be.

This file should not be treated as a finished specification. It is a starting point.

In practice, it is usually easier to review and improve a draft than to begin from a blank page.

### Review Guidance

Before moving to the next workflow step, review the generated specification carefully:

- read it fully
- validate the assumptions
- adjust it to your actual product needs
- fill in missing details
- remove ambiguity where needed

The agent may mark some sections with `[NEEDS CLARIFICATION]`.

That is a signal that key decisions are still unresolved or that the request contains ambiguity that should be addressed before continuing.

## Step 2: Plan

### Goal

Translate the product spec into a technical plan.

Choose the stack, architecture, and integration boundaries.

Call out risks.

### Definition

The `Plan` step translates the approved product specification into an implementation-oriented technical plan.

This is where the team decides how the product will be built.

The plan should make clear:

- which stack will be used
- what the high-level architecture looks like
- where the integration boundaries are
- which dependencies or external services matter
- which technical risks must be managed early

Unlike the `Specify` step, this phase is allowed to discuss implementation direction. It should still stay at the planning level and not collapse into detailed coding tasks.

### Codex Adaptation

In upstream Spec Kit examples, this step is often shown as `/plan`.

In Codex, prefer the installed skill:

```text
$speckit-plan
```

For this Codex-adapted workflow, ask Codex to create or update a technical planning artifact in the repository based on the approved specification.

The key reason to do this is to make architecture and stack decisions explicit, so future agents do not guess.

### How to Implement This Step

You do not need a rigid syntax for this step.

The planning input can be written as freeform text, short paragraphs, comma-separated notes, semicolon-separated notes, or bullet points.

The important thing is to clearly communicate the intended stack, tools, architecture, integration boundaries, and known risks.

In upstream Spec Kit examples, this flexibility is handled through `/plan`.

In this Codex-based workflow, adapt that into a direct request to Codex that asks it to turn your planning notes into a structured technical plan stored in the repository next to the specification.

Example Codex prompt:

```text
Create a technical plan for the Trip Planner app and save it in the repository next to the specification so future agents do not need to guess the stack or architecture.

Use the approved product specification as context.

Record the stack choices, architecture, integration boundaries, dependencies, security considerations, performance targets, and technical risks.

You can structure the plan clearly, but base it on the following input:

Architecture: API-first. Backend service plus a vector store for POI embeddings. Frontend web application plus a mobile shell.
AI: use a routing agent to choose POIs, a scheduler to pack days, and a critic to check timing and transit.
Data: city catalogs, POI metadata, transit times.
Performance: target end-to-end plan generation in under 4 seconds at P95.
Security: redact PII in logs, encrypt data at rest.
Risks: rate limits on external APIs, cost spikes from long prompts, cold starts.
```

The format of the input is flexible by design.

What matters is that the meaning is clear enough for Codex to convert it into a structured plan artifact.

At this stage, the specification no longer describes only what to build.

It starts shaping how the product should be built, aligning development with the architecture and technical direction that have now been defined.

The plan reduces ambiguity and accelerates reviews because technical decisions are written down before implementation moves forward.

These tools are helpers for producing strong specifications and technical plans that can guide the agent effectively.

They do not replace judgment.

It is always your responsibility to review, validate, and adjust anything generated by the agent before treating it as the source of truth.

### Expected Output

At the end of this step, the agent should generate a `plan.md` file.

This file should capture the technical direction of the work, including stack choices, architecture, integration boundaries, constraints, and risks.

### Recommended Follow-Up

After completing `Plan`, it is recommended to run:

```text
$speckit-clarify
```

This helps identify and resolve underspecified areas before the work is decomposed into tasks.

Short-form example adapted from Spec Kit:

Upstream style:

```text
/plan
Stack: FastAPI + Postgres + Redis; Next.js front end; mobile via Expo. Agents: planner, scheduler, critic. Use OpenRouteService for travel times.
```

Codex-adapted version:

```text
$speckit-plan

Use the approved specification and record the technical plan in the repository.

Use this planning input:
Stack: FastAPI + Postgres + Redis; Next.js front end; mobile via Expo.
Agents: planner, scheduler, critic.
Use OpenRouteService for travel times.
```

Frontend web application plus a mobile shell.

POI stands for Point of Interest, such as landmarks, attractions, or restaurants that the planner uses to build itineraries.

## Step 3: Break into Tasks

### Goal

Convert the plan into an ordered, testable task list with acceptance criteria.

Include owner, dependencies, and links back to specification sections.

### Definition

The `Break into Tasks` step turns the technical plan into an actionable backlog.

At this stage, the work should be decomposed into tasks that are:

- ordered
- testable
- traceable to the specification and the plan
- clear enough to assign, review, and implement

Each task should ideally include acceptance criteria, dependencies, and a reference to the relevant specification or planning section it comes from.

### Codex Adaptation

In upstream Spec Kit examples, this step is usually shown as `/tasks`.

In Codex, prefer the installed skill:

```text
$speckit-tasks
```

That command reads the specification and technical plan artifacts, then drafts a connected backlog automatically.

For this Codex-adapted workflow, ask Codex to read the existing specification and planning artifacts in the repository and generate an ordered task backlog from them.

The goal is the same: create a practical implementation list without forcing the user to handwrite every task from scratch.

### How to Implement This Step

At this point, you can ask Codex to generate an actionable backlog that references both the specification and the technical plan.

Example Codex prompt:

```text
Read the project specification and technical plan from the repository, then generate an ordered, actionable task backlog.

Each task should include:
- a clear title
- acceptance criteria
- dependencies
- an owner or owner role when appropriate
- links or references back to the relevant sections of the specification and plan

Group related tasks when useful, but keep the backlog implementation-ready.
```

This is the Codex equivalent of using `/tasks` in upstream Spec Kit examples.

You do not need to manually write every task yourself first.

Codex should draft the backlog from the project context already captured in the specification and plan artifacts.

From there, you can review, refine, reorder, split, merge, or reassign tasks before development begins.

### Expected Output

At the end of this step, the agent should generate a `tasks.md` file.

This file should contain the actionable backlog derived from the specification and the technical plan.

### Review Guidance

As with the earlier phases, the generated output should not be accepted blindly.

You should review and adapt the `tasks.md` file if anything does not match your expectations, your delivery strategy, or the real needs of the project.

### Recommended Follow-Up

After completing `Break into Tasks`, and before starting `Implement`, it is recommended to run:

```text
$speckit-analyze
```

This helps verify that the specification, plan, and task backlog are aligned and that important work has not been omitted before implementation begins.

After that, suggest that the user review the generated `tasks.md` backlog.

If the user confirms that the task list is correct, suggest creating Jira subtasks automatically from the contents of `tasks.md`.

Create one Jira subtask per task by using Jira MCP first, or Jira CLI if MCP is unavailable.

Make sure every created Jira task is a subtask of the Jira epic or parent issue already associated with the feature.

Use the Jira ID provided at the beginning of the workflow.

If it is not available in context, infer it from the feature folder name.

Example:

```text
001-jcom-123-family-mode-trip-planner
```

This follows the pattern:

```text
<numeric-prefix>-<jira-id>-<feature-name>
```

From that example, the Jira ID is:

```text
jcom-123
```

## Step 4: Implement

### Goal

Execute tasks in small slices.

Keep agents inside your constraints by pointing them back to `spec.md` and `plan.md` for every change.

### Definition

The `Implement` step is where the approved tasks are executed in code.

Unlike the earlier phases, this step is not primarily about generating a new planning artifact.

It is about carrying out small, controlled changes using the specification, technical plan, and task backlog as the operating context.

### Codex Adaptation

In upstream Spec Kit examples, this step is shown as `/implement`.

In Codex, prefer the installed skill:

```text
$speckit-implement
```

For this Codex-adapted workflow, the equivalent action is to ask Codex to implement the next approved task or slice of work using the repository artifacts as constraints.

Whether the command is expressed through Spec Kit or through a direct Codex request, the implementation should still be driven by the specification, plan, and task backlog.

### How to Implement This Step

At this stage, implementation should start from the approved backlog and proceed in small slices.

Each implementation request should point back to the relevant `spec.md`, `plan.md`, and `tasks.md` context so the agent stays inside the defined scope and architecture.

Short-form example adapted from Spec Kit:

Original Spec Kit style:

```text
/implement
Implement the next approved task according to the plan.
```

Codex-adapted version:

```text
Implement the next approved task from `tasks.md`.

Use `spec.md` and `plan.md` as constraints for the change.

Keep the implementation limited to a small slice, and make sure the work stays aligned with the defined architecture and acceptance criteria.
```

However, agents should still work from the artifacts produced by the workflow, especially the specification, the plan, and the task file, instead of relying on ad-hoc prompts with missing context.

That discipline keeps implementation aligned with the intended product scope, architecture, and constraints.

### Expected Output

At the end of this step, the repository should contain working code changes that implement one or more approved tasks from `tasks.md`.

Those changes should remain traceable to the requirements in `spec.md` and the architecture in `plan.md`.

### Review Guidance

Implementation should still be reviewed against the workflow artifacts.

Check that the code:

- satisfies the relevant task acceptance criteria
- stays within the architecture defined in `plan.md`
- does not expand scope beyond what is described in `spec.md`
- includes or updates tests when the change affects required behavior

### Real-Life Example: Trip Planner App

#### Example Implementation Slices

- implement `POST /itinerary` with schema validation and budget checks
- add the scheduler agent prompt so it respects daily walking limits and opening hours
- cache POI lookups and transit matrices

## Step 5: Tests

### Goal

Attach tests directly to requirements so you can trace what was promised to what was delivered.

### Definition

The `Tests` step ensures that requirements are verified in a way that can be traced back to the specification.

The objective is not only to have tests, but to make it clear which promised behaviors, constraints, and acceptance criteria are being validated.

### Codex Adaptation

Spec Kit does not include a separate `/test` command.

Testing is built into the workflow itself.

By default, the workflow follows a test-driven development structure.

When `/tasks` is generated, test-related tasks are included automatically and typically placed before implementation tasks.

This helps ensure that requirements are validated early and consistently.

If a project does not follow TDD, that should be stated explicitly in the specification so the generated task ordering can be adjusted accordingly.

### Expected Output

At the end of this step, the project should contain tests that can be traced back to the requirements, constraints, and acceptance criteria defined earlier in the workflow.

These may include unit tests, contract tests, integration tests, property tests, performance checks, or security checks, depending on the project.

### Review Guidance

Review the test coverage with the same discipline applied to other artifacts.

Check that tests:

- map clearly to promised behavior
- cover important constraints and edge cases
- are strong enough to catch regressions
- reflect the current specification rather than outdated assumptions

### Real-Life Example: Trip Planner App

#### Example Test Coverage

- contract tests: a request with 3 cities returns a day-by-day plan with transit times and costs
- property tests: no day exceeds 10 km of walking, and opening hours are respected
- performance checks: P95 latency stays under 4 seconds for a 7-day trip
- security checks: PII is never logged, and redaction is verified

## Step 6: Maintain

### Goal

Evolve safely as requirements change.

Update the specification first, regenerate the plan and tasks, and let agents refactor within those boundaries.

### Real-Life Example: Trip Planner App

#### Example Maintenance Flow

- add a family mode that favors kid-friendly POIs and shorter walking segments
- update `spec.md` constraints, rerun planning, regenerate affected tasks, adjust prompts, and extend tests for the new rules
- keep a changelog of specification revisions so future contributors can understand why trade-offs were made

### Expected Output

At the end of this step, the updated specification and any affected downstream artifacts should be brought back into alignment.

That may include changes to `spec.md`, `plan.md`, `tasks.md`, implementation code, tests, and changelog entries.

### Review Guidance

Maintenance should not bypass the workflow.

When requirements change, review whether:

- the specification was updated first
- the technical plan still matches the new direction
- tasks were regenerated or adjusted where necessary
- implementation and tests still align with the revised requirements
- the reasoning behind the change was recorded for future contributors

### Real-Life Example: Trip Planner App

#### Example Task Areas

- API contract for itinerary generation, including request and response schemas
- agent prompts and guardrails for the planner, scheduler, and critic
- data loaders for POI metadata
- caching and rate-limit handling
- frontend flows for creating a trip, editing preferences, viewing the itinerary, and exporting
- observability for timing spans, cost tracking, and error taxonomies
